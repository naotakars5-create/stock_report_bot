# 日本株 朝のスクリーニング速報（東証スクリーニング版）

東証上場銘柄を**機械的な条件でスクリーニング**し、出来高・移動平均・相対強度・過熱感
などから「**今日見るべき銘柄**」を整理して、朝の情報整理に使うためのレポートです。
ターミナルに出力 ＋ **LINEへ自動送信**し、**GitHub Actions で平日（月〜金）の朝8:15（日本時間・寄り付き前）に自動実行**できます（土日・祝日・年末年始は配信しません）。
銘柄一覧は **JPX公式の上場銘柄一覧(`data_j.xls`)を起動時に自動取得**するため、
手動でのCSV差し替えは不要です（取得に失敗しても既存CSVで継続します）。

> ## ⚠️ 本サービスの位置づけ（重要）
>
> - **これは投資助言ではありません。** AIが売買判断をするものでもありません。
> - **個別銘柄の「売買推奨」ではありません。** 「おすすめ株」「買い候補」を出すサービスではありません。
> - 公開データ（`yfinance`）をもとにした**機械的なスクリーニング結果**を、
>   情報整理の材料として提供するツールです。
> - 提示するのは「おすすめ株」ではなく「**スクリーニング上位銘柄**」「**本日の注目銘柄**」です。
> - 最終的な投資判断は、必ずご自身の責任で行ってください。

**株価データの取得にAPIキーは不要**です（`yfinance` の公開データを使用）。
LINE送信を使う場合のみ、LINEの認証情報を環境変数で設定します
（未設定ならターミナル表示だけで動作します）。

## 機能概要

1. **JPX公式の上場銘柄一覧(`data_j.xls`)を自動取得**して
   `jpx_listed_companies.csv` を更新し、そこから**普通株ユニバース**を構築
   - 起動時に最新の一覧をダウンロード（直近取得から20時間以内なら省略）
   - ダウンロードに失敗しても既存の `jpx_listed_companies.csv` で続行
   - `市場・商品区分(market)` に「内国株式」を含む行のみ採用
   - ETF・ETN / REIT・インフラファンド / 外国株 / 出資証券 を除外
   - 優先株など証券コードが4桁でないものを除外
   - 流動性が低すぎる銘柄（直近20日平均出来高が下限未満）を除外
   - yfinance用ティッカーは「コード + `.T`」で生成（例: 7203 → 7203.T）
2. 市場指数・為替を取得
   - 日経平均 / TOPIX / ドル円 / S&P500 / NASDAQ
3. **2段階スクリーニング**で重い処理を抑えつつ候補を絞り込み
   - **一次スクリーニング（軽量・高速）**: 全候補から上位50銘柄へ
     - 25日移動平均より上にある／5日線が25日線を上回る／出来高が増えている／急騰しすぎていない
     - データ取得できない銘柄はスキップ
   - **二次スクリーニング（詳細）**: 上位50銘柄を **7つの評価軸** で精密スコアリングし
     **スクリーニング上位5銘柄** を抽出（事業内容・テーマ性・流動性・バリュエーション・
     前回継続性を加味。評価点 6.0未満は除外）
4. 各銘柄を **8つの評価軸** でスコアリング（合計10.0点、小数第1位まで）
   - トレンド 1.7 / 出来高・流動性 1.3 / 相対強度 1.3 / 業種テーマ性 1.3 /
     **ニュース環境・マクロテーマ 1.0** / バリュエーション 0.9 /
     安定性・過熱回避 1.3 / 前回検証・継続性補正 1.2
   - **ニュース環境・マクロテーマ**は、世界情勢・経済ニュース（RSS）と市場データ（為替・
     米国市場）から市場テーマを抽出し、銘柄のテーマタグと照合して評価します（詳細は後述）
   - 相対強度の基準は、日経平均 `^N225` が異常値を返すことがあるため
     **TOPIX連動ETF(1306.T)** を市場平均として使用（詳細は後述）
5. レポートに以下を日本語で出力
   - 市場概況（日経平均 / TOPIX / ドル円 / S&P500 / NASDAQ ＋ 市場コメント）
   - 本日のスクリーニング結果（分析対象数 / 取得成功数 / 一次通過数 / 最終抽出数）
   - **スクリーニング上位5銘柄** を機械的に評価
     （順位 / 銘柄名 / 証券コード / 評価点・評価ランク / **今日見る理由** / **事業内容** /
     **テーマタグ** / **ニュース環境** / **評価グラフ（相対スコア7軸）** /
     **加点理由 / 減点理由** / **スコア内訳** / **リスクメモ**）
     ※「想定上昇率」「上昇予想」などは出しません
   - **今日の世界情勢・経済ニュース**（主要見出し / 市場テーマ / 日本株への影響メモ）と
     **今日強いテーマ／テーマ別スクリーニング上位**、**テーマ別の見方**
   - **検証結果**（過去の上位5銘柄を 前回・3営業日前・1週間前 で追跡。最大5勝5敗）
   - 末尾に注意書き（売買推奨ではない旨）
6. LINE配信は **「カード中心」** の構成です。**①サマリーカード（Flex）→ ②銘柄カルーセル
   （横スライドカード）→ ③短い補足テキスト** の順で配信（画像ホスティング不要）。
   毎朝 **1分で全体像が分かり、銘柄詳細はカードを横スライドして読む** ことを重視し、
   **長い銘柄別詳細テキストは送りません**（カードと重複させない）。
   - **①サマリーカード（トップページ）**: 1分で全体像がつかめる構成。
     **【今日の相場判定】（★5段階・国内指数／前日米国／物色の裾野から機械判定）** /
     スクリーニング集計 / 市場概況（日経平均・TOPIX・ドル円・S&P500・NASDAQ） /
     今日の主要テーマ＋**【今日最重要テーマ】の理由（なぜそのテーマか・100字以内）** /
     今日強いテーマ / **【本日のスクリーニング傾向】（機械的な観察の箇条書き）** /
     上位5銘柄の簡易ランキング / 検証結果の簡易サマリー（前回上位5・最大5勝5敗）/
     **【今日の結論】（★5段階・一言）** / 注意書き
   - **②銘柄カルーセル（銘柄詳細はここに集約）**: 上位5銘柄を **1銘柄＝1枚のカードで横スライド表示**
     （Apple風ミニマル・余白多め・アイコン＋色分け＋箇条書きで読みやすく）。各カードに
     銘柄名 / 証券コード / 業種 / **総合評価点（大きく表示）＋★5段階** / 評価ランク（High Fit〜Neutral）/
     事業・テーマ・ニュース（各1行） /
     **📊 需給・資金（機関投資家視点・機械的推定：出来高の読み／短期・中長期資金／需給）** /
     **📅 決算・イベント（取得可能な決算日・配当権利日のみ。市場予想比はデータ未対応と明示）** /
     **📈 テクニカル節目（参考：下値メド／上値メド／直近レンジ・移動平均と直近高安から算出）** /
     **📐 評価グラフ（Flex横棒・相対7軸）** / **⭐ スクリーニング適合度（★＋理由）** /
     **⚠️ リスク** / 売買推奨ではない注意書き を掲載
     （評価点に応じて 8.5+＝深い青系・7.5〜8.4＝青緑系・7.4以下＝グレー系の配色）。
     **銘柄ごとの情報は各カード内で完結**させ、重複を排除。補足テキストでは繰り返しません。
     カルーセルが50KBに近づくと**評価グラフを自動で省いて確実に配信**します（サイズ保護）。
   - **③短い補足テキスト**: カードと重複しない情報だけを載せます。
     **今日のニュース・マクロ環境 / 今日強いテーマ / テーマ別確認銘柄 / 検証結果 / 注意書き**
     のみ。**最大1500字（目安1000字以内）** で、LINEで一〜二画面に収まる長さです。
     各銘柄の評価グラフ・加点/減点理由・ニュース環境・リスクメモといった重複文は載せません。
   - **長文の詳細レポート（`report_writer.build_report()`）はLINEには送りません**。
     ターミナル確認および将来のWeb版／PDF版用に残しています。
   - Flexメッセージの送信に**失敗した場合のみ**、上位銘柄の概要を1行ずつ補った
     **短縮テキスト（`build_fallback_text`）へ自動フォールバック**します
     （フォールバック時も長文にせず、最大1500字に収めます）。
   - **詳細な検証（銘柄ごとの騰落）は履歴CSV（`data/report_history.csv`）で確認できます。**

## スクリーニング条件と評価軸

本Botは「将来上がる株」を予測するものではなく、**現時点のデータが一定の条件を
満たしているか**を機械的に判定して並べ替えます。

### 一次スクリーニング（軽量・高速に候補を絞る）

以下を**すべて**満たす銘柄が通過します。

- 流動性: 直近20日平均出来高が下限以上（`MIN_AVG_VOLUME`）
- 株価が25日移動平均より上にある
- 5日移動平均が25日移動平均を上回っている
- 直近出来高が増えている（5日平均 > 25日平均）
- 短期で過熱しすぎていない（直近5日の上昇率が閾値未満）

### 二次スクリーニング（7つの評価軸・合計 10.0点）

通過銘柄を次の7軸でスコアリングし、**評価点 6.0以上の上位5銘柄**を抽出します。

> **なぜ上位5銘柄だけを配信するのか**: 全銘柄（約3,700）を評価したうえで、朝に短時間で
> 目を通せるよう **最も条件が揃った5銘柄に絞って** 配信します。件数を絞ることで1銘柄ごとに
> 事業内容・評価バランス図・リスクメモまで載せたカードを用意でき、情報の取捨選択にかかる
> 手間を減らせます（売買推奨ではなく、朝の情報整理のための機械的な抽出です）。

| 評価軸 | 配点 | 見るもの |
|--------|------|----------|
| トレンド | 1.7 | 5日・25日・75日移動平均の並び、株価との位置（5日>25日>75日なら高評価） |
| 出来高・流動性 | 1.3 | 直近出来高/20日平均の増加、売買代金（流動性が低い銘柄は減点） |
| 相対強度 | 1.3 | 市場平均（TOPIX連動ETF）に対して強いか |
| 業種テーマ性 | 1.3 | 半導体・防衛・AI 等のテーマタグ（注目テーマはやや加点） |
| ニュース環境・マクロ | 1.0 | 世界情勢・経済ニュース＋為替/米国市場から抽出した市場テーマと、銘柄テーマの関連 |
| バリュエーション | 0.9 | PER / PBR / 配当利回り（取得できない場合は中立） |
| 安定性・過熱回避 | 1.3 | 短期の急騰しすぎ・高ボラティリティを減点 |
| 前回検証・継続性補正 | 1.2 | 前回も上位なら加点、ただし短期過熱が強ければ減点（前回なしは中立） |

**評価点の意味（売買判断ではなく抽出の目安）**

| 評価点 | 意味 |
|--------|------|
| 9.0以上 | 条件がかなり揃っている |
| 8.0〜8.9 | 複数条件で強い |
| 7.0〜7.9 | 一部条件が良い |
| 6.0〜6.9 | 監視候補 |
| 6.0未満 | 今回は除外 |

> 相対強度の基準は、日経平均 `^N225` が `yfinance` で異常値を返すことがあるため、
> 安定して取得できる **TOPIX連動ETF(1306.T)** を市場平均として使用しています。
> バリュエーション（PER/PBR/配当）は `yfinance` から best-effort で取得し、
> 取得できない銘柄は中立扱いにして全体を止めません。

### 総合評価バー・評価ランクの見方

各銘柄カードのヘッダーに総合評価（0〜10点）を**大きく表示**し、10ブロックのバーと
**評価ランク**を併記します。

```
8.6 / 10
SCREENING SCORE
█████████░
High Fit ・ 条件一致度が高い
```

| 評価点 | ランク（英語）| 日本語 | カード配色 |
|--------|---------------|--------|------------|
| 8.5以上 | High Fit | 条件一致度が高い | 深い青系 |
| 7.5〜8.4 | Good Fit | 複数条件が一致 | 青緑系 |
| 6.5〜7.4 | Watch | 監視候補 | グレー系 |
| 6.4以下 | Neutral | 中立 | グレー系 |

> ランクは「条件一致度」を表すもので、「買い」「強気」などの投資判断ではありません。

### 評価グラフ（評価バランス図）の見方

各銘柄カードでは、評価を **Flex Message の横棒グラフ**（薄いグレーのトラック＋色付きバー＋数値）で
表示します。**カード内**と**詳細テキスト**の両方に掲載します。

```
（カード）            （詳細テキスト）
トレンド  ███░ 4.2    トレンド■■■■□ 4/5
出来高    ██░░ 3.8    出来高　■■■■□ 4/5
相対強度  ████ 4.5    相対強度■■■■■ 5/5
テーマ    ████ 4.0    テーマ性■■■■■ 5/5
ニュース  ████ 4.3    ニュース■■■■□ 4/5
割安      ██░░ 2.8    割安感　■■■□□ 3/5
安定      ██░░ 3.2    安定性　■■■□□ 3/5
```

- バーは **候補全体（二次スクリーニングを受けた銘柄群）の中での相対スコア**です。各軸の値を
  平均・標準偏差で標準化し、平均を中央（2.5/5付近）に置いて強弱を広げています。
  これにより、上位銘柄でも **強い軸と弱い軸（2.0〜3.0台）が一目でわかる** ようにしています。
  （総合スコアは絶対評価のまま維持し、グラフの見え方だけ相対化しています。恣意的な操作はしません。）
- カードでは Flex の box でバーを描画し、テキスト（■□）よりカードデザインに馴染ませています。
  Flex送信に失敗した場合は、テキストの評価バランス図（詳細テキスト）にフォールバックします。
- 図には主要7軸（トレンド／出来高／相対強度／テーマ性／**ニュース**／割安感／安定性）を表示し、
  継続性補正は「スコア内訳」に含めて表示します。

### 今日見る理由・加点理由・減点理由

各銘柄カードには、点数の納得感を高めるために次を表示します。

- **今日見る理由**（カード上部）: なぜ今日この銘柄が上位なのかを1文で要約。事業・テーマ・テクニカル・
  出来高のうち最低2要素を含めます。例:「防衛・重工テーマとの接点があり、出来高増を伴って短期トレンドが強い。」
- **加点理由**（2〜3個）: 相対的に強い評価軸とニュース接点を、数値を添えて表示。
- **減点理由**（1〜2個）: 相対的に弱い評価軸・リスク・注意点を表示。無い場合は「大きな減点要素は限定的」。

```
今日見る理由：半導体・AIテーマとの接点があり、出来高増を伴って短期トレンドが強い。

加点理由：
＋ 短期・中期トレンドが上向き（5日線が25日線を+1.8%）
＋ 出来高が20日平均を上回る（約1.5倍）
＋ 半導体・AIタグが、半導体・NASDAQ関連のニュースと関連
減点理由：
－ 安定性スコアが低い（日次ボラ2.8%）
－ バリュエーションは割高感がある
```

> 「買い」「上昇予想」「強気」などの売買推奨表現は使いません。あくまで点数の根拠を示すものです。

## ニュース環境・マクロテーマ評価（世界情勢の取り込み）

株価データだけでなく、**世界情勢・経済ニュース・為替・米国市場**などのマクロ環境を評価に
取り込みます。`news_fetcher.py`（見出し取得）と `macro_analyzer.py`（テーマ分析）が担当します。

### 使用するニュースソース（APIキー不要）

- **Google ニュース RSS**（ビジネス/経済トピック、および 日経平均・為替・米国株・金利・原油・
  半導体・防衛 を横断する検索クエリ）
- 見出しのみを取得し、RSSの解析は Python標準ライブラリ（`xml.etree`）で行います。
- **APIキーは不要**です（GitHub Actions でもそのまま動作します）。

### 評価の考え方（銘柄ごとに固有のコメント）

1. 見出しから 30以上の市場テーマ（半導体・AI・データセンター・防衛・地政学リスク・円安/円高・
   金利・銀行/保険・商社・資源・原油・電力・インバウンド・中国景気・NASDAQ・自動車・機械・重工・
   造船・建設・医療・物流・小売・不動産 など）を抽出し、強度を数値化
2. 為替（ドル円）や米国株（S&P500/NASDAQ）の方向感も加味し、`macro_context` を作成
   （`major_themes` / `positive_theme_tags` / `caution_theme_tags` / `market_summary` /
   `fx_comment` / `us_market_comment` / `rates_comment` / `commodity_comment` /
   `geopolitical_comment` / `raw_headlines`）
3. **銘柄ごと**に `theme_tags`・`sector`・`business_summary` と当日テーマを照合し、
   `calculate_macro_relevance_score()` で **ニュース関連スコア（0〜1）・関連理由・注意点**を算出
   - 例: 防衛ニュースが多い → 防衛・重工タグの銘柄にプラス
   - 例: 円安進行 → 輸出関連（円安メリット）にプラス、ただし円高局面は注意
   - 例: 原油高 → 資源・商社にプラス、運輸・電力は注意
   - 例: 金利上昇 → 銀行・保険にプラス、グロース・不動産は注意
4. `build_stock_news_comment()` が、その銘柄固有の「**ニュース環境**」コメント（2〜3文）を生成
   （カードは要約版、詳細テキストはやや詳しい版）。関連が弱い銘柄は「明確な個別テーマは限定的」と表示

> 表現は「意識されやすい」「関連がある」「注意」にとどめ、「買い」「買われる」「上がる」等は使いません。

### ニュース取得に失敗した場合の挙動

- `news_fetcher.py` が取得に失敗したら **空リスト** を返します。
- `macro_analyzer.py` は見出しが無くても落ちず、**ニュース評価は中立（0.5）** になります。
- レポートには「**ニュース取得が一部制限されています**」と表示し、**スクリーニングと LINE送信は
  そのまま継続**します（マクロ評価が一時的に効かないだけで、全体は止まりません）。

## 事業内容・テーマタグの管理（company_profiles.csv）

各銘柄の事業内容とテーマタグは、`profile_loader.py` が次の2段構えで用意します。

- **A. `company_profiles.csv` に登録がある銘柄**: その事業内容（`business_summary`）・テーマタグを使用。
  主要な大型・中型・テーマ株を中心に固有の事業内容を整備しています（順次拡充）。
- **B. 未登録の銘柄**: 業種名・テーマタグから**自然な事業説明を自動生成**します。
  「業種情報をもとにした簡易分類」のような無機質な表現は使わず、
  例えば「〇〇領域で事業を展開する企業（△△テーマと接点）」のように表示します。
- 事業内容に含まれるキーワードからもテーマタグを補完し（`tags_from_business`）、ニュース照合に活用します。

CSV に `business_summary` が登録されている銘柄は、カードでは短く、詳細テキストではそのまま表示します。

### `company_profiles.csv` の更新方法

列は次の通りです。気になる銘柄から手入力で増やしていけます。

```csv
code,name,sector,business_summary,theme_tags,size_category,notes
7011,三菱重工業,機械,防衛・宇宙・エネルギー関連の総合重工大手,防衛|重工|エネルギー,大型,
```

- `theme_tags` は `|`（または `;`・`、`）区切りで複数指定できます。
- 行を追加・編集して保存するだけで、次回実行から反映されます。

### theme_tags の使い方

テーマタグは「業種テーマ性評価」のスコアに使われ、レポートの「タグ：」にも表示されます。
扱うテーマ例: 半導体 / 防衛 / 電力 / 金融 / 商社 / インバウンド / AI / データセンター /
円安メリット / 資源 / エネルギー / 造船 / 重工 / 建設 / インフラ / 内需ディフェンシブ /
物流 / 医療 / DX / サイバーセキュリティ。
（半導体・防衛・AI・データセンター・サイバーセキュリティは注目テーマとしてやや加点）

## 対象銘柄数の設定（MAX_STOCKS）と全銘柄へ広げる方法

分析対象の最大銘柄数は `main.py` の `MAX_STOCKS`、または**環境変数 `MAX_STOCKS`** で設定します
（環境変数が優先）。**既定は「全銘柄」（`MAX_STOCKS = None`・制限なし）**で、`MAX_STOCKS` に
数値を設定したときだけ、その件数（コード昇順で先頭から）に絞り込みます。デバッグ用です。

```powershell
# 例: 一時的に300銘柄で実行
$env:MAX_STOCKS = "300"; python main.py

# 全銘柄（約3,700）を対象にする
$env:MAX_STOCKS = "all"; python main.py   # none / 0 でも全銘柄
```

`main.py` の値を直接変えてもOKです（`MAX_STOCKS = None` で全銘柄）。

> ⚠️ 全銘柄を対象にすると一次スクリーニングのデータ取得に**十数分〜数十分**かかります。
> まずは 300〜500 から始め、所要時間を確認しながら増やしてください。
> GitHub Actions では `timeout-minutes: 60` を設定済みです。

## 検証機能（過去のスクリーニング上位5銘柄の追跡）

毎回の **上位5銘柄** を `data/report_history.csv` に保存し、過去の上位銘柄がその後どう動いたかを
機械的に集計して表示します。**前回（直近）に加え、3営業日前・1週間前（5営業日前）** の回も、
データがある範囲で追跡します。

- 保存項目: 実行日 / 証券コード / 銘柄名 / 順位 / 評価点 / 実行時株価 / 業種 / テーマタグ
- 検証は **その回の上位5銘柄だけ** を対象にするため、**勝敗数は必ず最大5件**（例: 3勝2敗）です。
- 同じ実行日に複数回実行しても、**最後の1回だけ** を残します（同日重複は排除）。
- 検証で表示する内容:
  - 上位5銘柄の平均騰落率
  - **日経平均比**（取得できれば TOPIX比 も）
  - 上昇 / 下落の数（◯勝◯敗、最大5件）
  - 最も上昇した銘柄 / 最も下落した銘柄
  - 前回実行日
  - 現在株価が取得できない銘柄がある場合は「**取得可能な◯銘柄のみで検証**」と表示
- データがまだ無い場合は「**検証データは蓄積中です**」と表示します。

```
■ 検証結果（過去のスクリーニング上位5銘柄の追跡）
  前回（2026-06-26）上位5銘柄：3勝2敗・平均騰落率 +0.8%
    日経平均比 +0.3pt・TOPIX比 +0.4pt
  3営業日前（2026-06-24）上位5銘柄：4勝1敗・平均騰落率 +1.6% …
  1週間前（2026-06-20）上位5銘柄：2勝3敗・平均騰落率 +2.1% …
```

> これは**過去の抽出結果の追跡**であり、運用成績や将来の成果を保証・示唆するものでは
> ありません。GitHub Actions では実行環境が毎回リセットされるため、ワークフローが
> `data/report_history.csv` をリポジトリへコミットして永続化します
> （`permissions: contents: write` を使用）。多期間の検証には履歴の蓄積が必要なため、
> 毎営業日の実行を続けるほど検証が充実します。

## LINE配信時の注意点

- LINE配信は **「カード中心」** です。「**①サマリーカード → ②銘柄カルーセル（横スライドカード）
  → ③短い補足テキスト**」の順で送信します（カードは LINE Flex Message。画像ホスティングは不要）。
  **有料配信を想定し、長いほど良いのではなく、短時間で価値が伝わる構成**にしています。
- サマリーカードの先頭に **【今日の相場判定】（★5段階）**、末尾に **【今日の結論】（★5段階）** を置き、
  途中に **【今日最重要テーマの理由】** と **【本日のスクリーニング傾向】** を掲載。1分で全体像がつかめます。
- **銘柄ごとの詳細はすべて横スライドカード内に集約**します（Apple風ミニマル・アイコン＋色分け＋箇条書き）。
  上位5銘柄を 1銘柄＝1枚のカードで横にスワイプして見られ、各カードに 銘柄名 / 証券コード / 業種 /
  総合評価点（大きく表示）＋**★5段階** / 評価ランク（High Fit〜Neutral）/ 事業・テーマ・ニュース（各1行）/
  **📊 需給・資金（機関投資家視点・機械的推定）** / **📅 決算・イベント（日程のみ・予想比は未対応と明示）** /
  **📈 テクニカル節目（参考・下値/上値メド）** / **📐 評価グラフ（相対7軸）** /
  **⭐ スクリーニング適合度（★＋理由）** / **⚠️ リスク** / **売買推奨ではない注意書き** を掲載します。
  重複は排除し、情報密度を上げています（50KB接近時は評価グラフを自動省略して確実に配信）。
- **補足テキストはニュース・テーマ・検証結果のみ**です。カードと重複する銘柄別の評価理由・
  評価グラフ・ニュース環境・リスクメモは**繰り返しません**。内容は
  **今日のニュース・マクロ環境 / 今日強いテーマ / テーマ別確認銘柄 / 検証結果 / 注意書き** で、
  **最大1500字（目安1000字以内）**・LINEで一〜二画面に収まる長さに整えています。
- **長文の詳細レポートはLINEでは送りません**（`report_writer.build_report()` はターミナル確認・
  将来のWeb版／PDF版用に残しています）。
- **Flexメッセージの送信に失敗した場合のみ**、上位銘柄の概要を1行ずつ補った
  **短縮テキスト（`build_fallback_text`）へ自動フォールバック**します（最大1500字に収めます）。
- **詳細な検証（銘柄ごとの騰落）は履歴CSV（`data/report_history.csv`）で確認できます。**
- 環境変数（`LINE_CHANNEL_ACCESS_TOKEN` / `LINE_USER_ID`）が未設定の場合は LINE送信をスキップし、
  ターミナル表示のみで完結します。
- 長いテキストはLINEの文字数制限（1通5000字）に配慮して自動分割送信します。
- すべての配信に「**売買推奨ではありません**」という主旨の注意書きを必ず付けます。

## 表現に関するガイドライン（有料配信を行う場合の注意）

将来的に有料配信する場合も含め、**投資助言・売買推奨と誤解される表現は使わない**でください。

| 使わない表現 | 代わりに使う表現 |
|--------------|------------------|
| おすすめ株 / 買い候補 / 買い / 強気 | スクリーニング上位銘柄 / 今日見るべき銘柄 / 機械的な抽出結果 |
| 今後伸びる株 / 狙い目 / 上昇予想 | 今日見る理由 / トレンド評価 / 出来高評価 / 相対強度 / テーマ接点 |
| 想定上昇率 / 儲かる / 勝てる / 利益が出る | 加点理由 / 減点理由 / ニュース環境 / 過熱感 / リスクメモ / 検証結果 / 市場比 |
| 押し目買い価格 / 利確目安 / 損切りライン | **テクニカル節目（下値メド／上値メド／直近レンジ・参考値）** |
| 強気 / 弱気 / リスクオフ（相場判定） | **今日の相場判定（リスク選好〜リスク回避の機械判定）** |
| 期待値 / 短期期待 / 中期期待 | **スクリーニング適合度（条件一致の強さ）** |
| なぜ今買われているか | **需給・資金（機械的推定）／出来高の読み** |

推奨表現の例: スクリーニング上位銘柄 / 今日見る理由 / 今日見るべき銘柄 / 機械的な抽出結果 /
評価理由 / 加点理由 / 減点理由 / トレンド評価 / 出来高評価 / 相対強度 / テーマ接点 /
**ニュース環境 / 需給・資金（機械的推定）/ テクニカル節目（参考）/ スクリーニング適合度 /
今日の相場判定（リスク選好〜リスク回避）** / 世界情勢 / 過熱感 / リスクメモ / 市場比 / 検証結果 /
売買推奨ではありません。
禁止表現: おすすめ株 / 買い候補 / 今後伸びる株 / 上昇予想 / 想定上昇率 / 狙い目 / 儲かる / 買い /
強気 / 勝てる / 利益が出る / 押し目買い / 利確 / 損切り。

- レポートはあくまで「**機械的なスクリーニング結果**」「**リスクメモ**」として提示します。
- 「**売買推奨ではありません**」を明記し、断定的・誘引的な表現は避けてください。
- 有料配信では特に、金融商品取引法上の「投資助言」に該当しないよう、
  個別銘柄の売買を勧める表現を避ける運用を徹底してください（必要に応じて専門家に確認）。
- **カードの分析（需給・資金／テクニカル節目／相場判定）はすべて「非投資助言」の枠**で、
  客観的な価格の節目・出来高・ボラティリティなどの**事実／統計**として提示します
  （`stock_insights.py` が生成。押し目買い・利確・損切り・強気/弱気は使いません）。

### データの正確性（捏造しない）

金融プロダクトとして、**実データが無い項目は「データ未対応」と明示**し、推測で埋めません。

- **取得する（best-effort）**: 決算予定日・配当権利落ち日（yfinance カレンダー・欠損あり）、
  価格／出来高から算出するテクニカル節目・需給推定・相対評価。
- **データ未対応（カードに明示）**: 決算の**市場予想比**（売上/利益/ガイダンスの◎〇△）、
  信用需給（信用買い残）、指数組入・株主総会・展示会・政策・IR 等の日程。
  → これらは取得元が無い／日本株では不安定なため、`★`評価を捏造せず未対応と表示します。

## 全銘柄版の仕組み（JPX公式の上場銘柄一覧を自動取得：方法B）

本Botは **JPX公式の上場銘柄一覧(`data_j.xls`)を起動時に自動取得**し、
`jpx_listed_companies.csv`（`code,name,market,sector`）に変換してから分析します。
**手動でのExcelダウンロード・CSV差し替えは不要**です。
既定の `MAX_STOCKS = None` で**東証の全普通株（約3,700銘柄）**が対象になります。

担当モジュールは `jpx_fetcher.py` で、`main.py` 冒頭の以下の設定で制御します。

| 設定 | 既定値 | 意味 |
|------|--------|------|
| `AUTO_FETCH_JPX` | `True` | 起動時に自動取得する（`False` で既存CSVのみ使用） |
| `JPX_CSV_MAX_AGE_HOURS` | `20` | 直近取得からこの時間内なら再ダウンロードを省略 |

### 動作の流れ

1. 起動時に `jpx_fetcher.ensure_jpx_csv()` が呼ばれる
2. 直近の取得が新しければ（`JPX_CSV_MAX_AGE_HOURS` 未満）ダウンロードを省略
   - 取得時刻は `jpx_listed_companies.csv.fetched_at`（gitignore済み）に記録
3. それ以外は <https://www.jpx.co.jp/.../data_j.xls> をダウンロードして
   `jpx_listed_companies.csv` を上書き更新
4. **ダウンロードや解析に失敗しても処理は止まらず**、
   既存の `jpx_listed_companies.csv`（前回取得分／同梱分）で続行する

> 💡 `data_j.xls` の読み込みには `xlrd` を使用します（`requirements.txt` に同梱）。
> JPXファイル単体での取得テストは `python jpx_fetcher.py` でも実行できます。

> ⚠️ **処理時間に注意**
> 全銘柄（約3,700銘柄）を対象にすると、一次スクリーニングのデータ取得だけで
> **十数分〜数十分**かかることがあります。処理時間を抑えたい場合は
> `main.py` の `MAX_STOCKS` に数値（例: `500`）を設定してください
> （コード昇順で先頭から指定件数に絞り込みます）。
> 一次スクリーニングは進捗（`進捗 NN/総数`）を表示します。

### フィルタリングの仕組み（自動で普通株のみに絞り込み）

`universe_loader.py` が以下を自動で行います。

- `市場・商品区分` に「内国株式」を含む行のみ採用
  → **ETF・ETN / REIT・インフラファンド / 外国株 / 出資証券 を自動除外**
- 証券コードがちょうど4桁の行のみ採用 → **優先株（5桁コード）などを除外**
- 一次スクリーニング時に **流動性が低すぎる銘柄**（直近20日平均出来高 < `MIN_AVG_VOLUME`）を除外

### 自動取得を使わない場合（方法A：ローカルCSV固定）

ネットワークを使わず手元のCSVだけで運用したい場合は、
`main.py` の `AUTO_FETCH_JPX = False` にしてください。
その状態では `jpx_listed_companies.csv` を編集・差し替えすれば、
それがそのまま分析対象になります（列見出しは日本語/英語どちらも自動認識）。

## カスタマイズ

### 候補銘柄ユニバースを変える

`jpx_listed_companies.csv` を編集してください。列は以下の通りです。

```csv
code,name,market,sector
7203,トヨタ自動車,プライム（内国株式）,輸送用機器
6758,ソニーグループ,プライム（内国株式）,電気機器
```

- `code` … 証券コード（**4桁**。`.T` は付けない。yfinance用ティッカーは自動生成）
- `name` … レポートに表示する銘柄名（日本語可）
- `market` … 市場・商品区分。「内国株式」を含む行だけが普通株として採用される
- `sector` … 業種（任意・現状は表示に未使用だが将来の拡張用）

### 各種しきい値・件数を変える

`main.py` 冒頭の `MAX_STOCKS` / `PRIMARY_TOP_N` / `FINAL_TOP_N` / `MIN_AVG_VOLUME` を編集してください。

### スコアリングの重み付けを変える

`stock_scorer.py` の `WEIGHTS` を編集すると、二次スクリーニングの各観点の配点を調整できます。
一次スクリーニングの急騰しきい値・流動性下限は `PRIMARY_MAX_SURGE_5D` / `PRIMARY_MIN_AVG_VOLUME` です。

## LINE送信のセットアップ

レポートを自分のLINEへ自動送信するには、LINE Messaging API のチャネルを作成し、
**チャネルアクセストークン**と**自分のユーザーID**を取得します。

### 1. LINE Developers で Messaging API チャネルを作る

1. [LINE Developers コンソール](https://developers.line.biz/console/) にLINEアカウントでログイン
2. **プロバイダー**を作成（例: 自分の名前など。すでにあれば流用可）
3. プロバイダー内で **「新規チャネル作成」→「Messaging API」** を選択
4. チャネル名・説明・カテゴリなど必須項目を入力して作成
5. 作成後、スマホのLINEで、そのチャネル（公式アカウント）を**友だち追加**しておく
   （友だちになっていないとPush送信が届きません）

### 2. チャネルアクセストークンを取得する

1. 作成したチャネルの **「Messaging API設定」** タブを開く
2. 一番下の **「チャネルアクセストークン（長期）」** で **「発行」** を押す
3. 表示された文字列が `LINE_CHANNEL_ACCESS_TOKEN` です（再表示できないので控える）

### 3. LINE_USER_ID（自分のユーザーID）を取得する

ユーザーIDは `U` から始まる文字列です。取得方法の例:

- **方法A（簡単）**: 「Messaging API設定」タブの中の **「あなたのユーザーID」** 欄に
  自分のIDが表示されます（チャネル管理者の場合）。
- **方法B（Webhookで確認）**: チャネルを友だち追加した状態でメッセージを送り、
  Webhookで受信した `events[].source.userId` を確認します。
- いずれの場合も、取得した `U...` の文字列が `LINE_USER_ID` です。

> 補足: ここで使う `LINE_USER_ID` は「友だちのユーザーID」であり、
> プロフィールに表示される「LINE ID（@から始まる検索用ID）」とは**別物**です。

#### 方法B-2: スクリプトで取得する（`get_line_user_id.py`）

「あなたのユーザーID」欄が見つからない場合は、付属の補助スクリプト
`get_line_user_id.py`（Flask製の簡易Webhookサーバー）を使うと確実に取得できます。
Botにメッセージを送ると、受信したWebhookから `userId` をターミナルに表示します。

> ⚠️ このスクリプトは **LINE_USER_ID を一度取得するためだけ**のものです。
> 本番の毎朝レポート（`main.py` / GitHub Actions）では一切使いません。
> 取得が終わったらサーバーは停止してください。

**① 依存をインストール（Flaskを含む）**

```powershell
.\.venv\Scripts\Activate.ps1
pip install -r requirements.txt
```

**② サーバーを起動する**

```powershell
python get_line_user_id.py
```

`Running on http://0.0.0.0:5000` と表示されれば起動成功です（ポート5000で待ち受け）。
このターミナルは**起動したまま**にしておきます。

**③ ngrok でローカルサーバーを外部公開する**

LINEのWebhookはインターネットからアクセスできるURLが必要なため、
[ngrok](https://ngrok.com/download) でローカルの5000番ポートを公開します。
**別のターミナル**を開いて実行します（初回はngrokの無料アカウント登録と
`ngrok config add-authtoken <あなたのトークン>` が必要です）。

```powershell
ngrok http 5000
```

表示される `Forwarding` の **`https://xxxxxxxx.ngrok-free.app`** のようなURLを控えます。

**④ LINE Developers の Webhook URL に設定する**

1. LINE Developersコンソールで対象チャネルの **「Messaging API設定」** を開く
2. **Webhook URL** に、③で控えたURLの末尾に **`/callback`** を付けて設定
   - 例: `https://xxxxxxxx.ngrok-free.app/callback`
3. **「検証(Verify)」** を押す → `成功` と出ればOK
   （このときサーバー側ターミナルには「イベント数: 0」と表示されます）
4. **「Webhookの利用」をオン**にする

> 補足: 同じ画面の「応答メッセージ」「あいさつメッセージ」はオフでも構いません。

**⑤ Botにメッセージを送って userId を取得する**

スマホのLINEで、友だち追加した公式アカウント（Bot）に**何かメッセージを送信**します。
すると `get_line_user_id.py` を起動したターミナルに次のように表示されます。

```
========================================================
Webhook を受信しました
イベント数: 1

[event 0] type=message source.type=user
  ┌────────────────────────────────────────
  │  LINE_USER_ID = U1234567890abcdef1234567890abcdef
  └────────────────────────────────────────
  → この値を .env または GitHub Secrets の LINE_USER_ID に設定してください。
========================================================
```

この `U...` が `LINE_USER_ID` です。控えたら、サーバーは **Ctrl+C** で停止し、
ngrok も停止して構いません（以降の毎朝レポートには不要です）。

### 4. ローカルで使う（.env）

```powershell
# テンプレートをコピー
Copy-Item .env.example .env
# .env をエディタで開き、トークンとユーザーIDを記入
```

`.env` の中身:

```
LINE_CHANNEL_ACCESS_TOKEN=（手順2のトークン）
LINE_USER_ID=（手順3のU...から始まるID）
```

この状態で `python main.py` を実行すると、ターミナル表示に加えてLINEにも届きます。
`.env` は `.gitignore` 済みなのでGitには含まれません。

> 環境変数が未設定の場合は、**LINE送信を自動でスキップ**してターミナル表示だけ行います。
> 送信に失敗しても、プログラム全体は止まりません（警告を表示して継続）。
> 長いレポートはLINEの文字数制限（1メッセージ5000文字）に配慮して**自動分割送信**します。

## 平日朝の自動実行（GitHub Actions）

`.github/workflows/daily_report.yml` により、**平日（月〜金）の朝8:15（日本時間・寄り付き9:00前）に自動実行**されます（土日・祝日・年末年始は配信しません）。

### 仕組み

- GitHub Actions の `schedule` (cron) でジョブを起動します。
- cron は **UTC** で指定するため、`15 23 * * 0-4`（UTC 23:15・日〜木）＝ **翌日 JST 08:15・月〜金** としています。
  （cron の曜日も UTC 基準なので、「UTC 日〜木」が「JST 月〜金」に対応します。）
- **休場日ガード**: `main.py` 冒頭で `market_calendar.is_market_open()` を判定し、
  **土日・祝日（`jpholiday`）・年末年始（12/31〜1/3）は配信せず正常終了**します
  （cron だけでは祝日を除外できないため。手動実行 workflow_dispatch でも同様にスキップ）。
- **データ鮮度ガード**: 配信直前に「取得データの基準日（前営業日の大引け）」と
  「いま時点で確定しているべき最新立会い日」を照合し、**データが古い場合は配信を中止**して
  管理者（`ADMIN_USER_ID`）にLINE通知します（前日と同一データの再配信を防止）。
- 寄り付き前の実行のため、データは**前営業日の終値ベース**です（当日場は未了）。
- 配信ヘッダーは**「データ基準日：◯月◯日 大引け時点」**と表示し、鮮度が分かるようにしています。
- ジョブ内で Python をセットアップ →依存インストール → `python main.py` を実行します。
- LINE認証情報は **GitHub Secrets** から環境変数として渡します。
- 手動でも実行できます（Actionsタブ →「Daily Stock Report」→「Run workflow」）。

> ⚠️ **許容遅延**: GitHubのcronは混雑状況により**数分〜数十分ずれて起動**することがあります（仕様）。
> 全銘柄の生成に約15〜20分かかるため、8:15起動なら通常 **8:35前後に配信**され寄り付き（9:00）前に
> 間に合う想定です。遅延が大きい日は9:00以降にずれ込む可能性があり、これは許容範囲としています。
> より確実に寄り付き前へ届けたい場合は起動時刻を早めてください（例: JST 8:00 = `0 23 * * 0-4`）。

### GitHub Secrets の設定方法

1. GitHubのリポジトリページ → **Settings** → **Secrets and variables** → **Actions**
2. **「New repository secret」** を押し、以下の2つを登録する

   | Name | Secret（値） |
   |------|--------------|
   | `LINE_CHANNEL_ACCESS_TOKEN` | LINEのチャネルアクセストークン |
   | `LINE_USER_ID` | あなたの `U...` から始まるユーザーID |
   | `ADMIN_USER_ID`（任意） | データ異常・配信中止時の通知先ユーザーID（未設定ならログのみ） |

3. ワークフローはこのSecretsを自動で読み込みます（コードに値を書く必要はありません）。

### テスト・運用モード

- `python main.py --dry-run` … LINEに**送信せず**、生成内容（補足テキスト等）の表示のみ。履歴CSVも更新しません。
- 環境変数 `TEST_USER_ID` を設定すると、配信先を**そのテストユーザーに限定**（本番 `LINE_USER_ID` には送りません）。
- 環境変数 `SIMULATE_DATE=YYYY-MM-DD`（テスト用）で「その日付の朝8:15に実行した」状況を再現できます
  （休場日スキップ・鮮度ガードの受け入れテスト用。本番では未設定）。

### 実行タイミングを変えたい

`daily_report.yml` の cron を編集してください（UTC指定）。例:

```yaml
on:
  schedule:
    - cron: "15 23 * * 0-4"  # JST 月〜金 08:15（既定・寄り付き前。土日は配信しない）
    # - cron: "0 23 * * 0-4"  # 平日 JST 08:00 にしたい場合（遅延に備え早める）
    # - cron: "15 23 * * *"   # 毎日 JST 08:15 にしたい場合（祝日は main.py がスキップ）
```

## トラブルシューティング

| 症状 | 対処 |
|------|------|
| `ModuleNotFoundError: yfinance` | `pip install -r requirements.txt` を実行 |
| JPXファイルの読み込みに失敗（`xlrd`） | `pip install -r requirements.txt` を実行（`xlrd>=2.0.1` が必要） |
| 「JPX自動取得に失敗」と表示される | ネットワーク／JPX側URL変更の可能性。既存CSVで自動継続します。`MAX_STOCKS` を絞るか時間をおいて再実行。手動更新は `python jpx_fetcher.py` |
| 銘柄一覧を毎回ダウンロードしてほしくない | `JPX_CSV_MAX_AGE_HOURS` を大きくする、または `AUTO_FETCH_JPX = False` で固定 |
| 全銘柄が「取得失敗」になる | ネットワーク接続を確認。時間をおいて再実行 |
| `Activate.ps1` が実行できない | 上記の `Set-ExecutionPolicy` を参照 |
| ターミナルで日本語が文字化けする | `main.py` 内で標準出力をUTF-8に再設定済み。それでも崩れる場合は `chcp 65001` を実行してから起動 |
| TOPIXが「TOPIX(連動ETF)」と表示される | TOPIX指数本体はyfinanceで安定取得できないため、連動ETF(1306.T)を代理指標として使用しています |
| 特定銘柄だけデータ不足でスキップ | 新規上場などで履歴が短い銘柄。一次/二次で必要日数に満たない場合は自動スキップされます |
| 処理が遅い・終わらない | `MAX_STOCKS` を小さくする（例: 100〜300）。全銘柄版は十数分以上かかることがあります |
| 一次スクリーニング通過が0件 | 相場全体が下落局面だと条件を満たす銘柄が減ります。`MIN_AVG_VOLUME` を下げる等で調整可能 |
| LINEに届かない（送信スキップ表示） | `LINE_CHANNEL_ACCESS_TOKEN` / `LINE_USER_ID` が未設定。`.env` またはSecretsを確認 |
| LINE送信が HTTP 400 で失敗 | `LINE_USER_ID` が誤り（`@`から始まる検索IDではなく`U...`のユーザーID）。チャネルを友だち追加済みかも確認 |
| LINE送信が HTTP 401/403 で失敗 | チャネルアクセストークンが誤り・失効。LINE Developersで再発行して設定し直す |
| GitHub Actionsが動かない | 既定ブランチに `daily_report.yml` がマージされているか、Actionsが有効か、Secretsが登録済みかを確認 |
| 一次/二次の通過数が想定より少ない | 一次は「25日線超え＋5日線>25日線＋出来高増＋非急騰＋流動性」を**すべて**満たす銘柄のみ通過します |
